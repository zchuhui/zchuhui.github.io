AI 模型选型指南

从任务类型、上下文、工具调用、成本和部署约束出发，建立可复用的模型选择方法。

更新于 2026年7月5日 模型选型LLM成本推理
目录

    先看任务，不先看榜单

    模型选型最常见的误区，是一上来问“哪个模型最强”。真实项目里更重要的问题是：

    • 任务是否需要强推理？
    • 是否需要读很长的上下文？
    • 是否需要工具调用或结构化输出？
    • 是否需要稳定低延迟？
    • 数据能不能出境？
    • 成本能不能被业务承受？

    一个模型在排行榜上很强，不代表它适合你的场景。选型的目标不是“用最强模型”，而是“用刚好能稳定完成任务的模型”。

    五类典型任务

    任务类型关键能力模型选择
    日常问答 / 摘要速度、成本、中文表达快模型、通用模型
    代码生成 / 调试指令跟随、代码上下文、工具调用编程优化模型
    复杂规划 / 架构推理深度、长上下文强推理模型
    RAG 问答忠实引用、低幻觉通用模型 + 好检索
    Agent 自动执行工具调用、错误恢复、长任务稳定性Agent 友好模型

    选型维度

    1. 质量

    质量不是“回答看起来聪明”，而是能否稳定满足你的业务标准。

    评估重点：

    • 事实是否正确
    • 是否遵守输出格式
    • 是否能处理边界情况
    • 是否会编造不存在的 API、字段、文件
    • 是否能基于工具结果修正判断

    2. 上下文长度

    长上下文适合：

    • 阅读大型代码库
    • 分析长合同、长报告
    • 多轮对话记忆
    • 一次性塞入大量日志

    但长上下文不是免费午餐。上下文越长，成本越高，噪声越多，模型越可能抓不住重点。很多时候，好的检索和摘要比盲目塞满上下文更可靠。

    3. 工具调用

    Agent 场景优先看工具调用能力：

    • 是否能稳定选择正确工具
    • 参数是否符合 schema
    • 工具失败后能否重试或换方案
    • 是否会循环调用同一工具
    • 是否能区分“工具事实”和“自己猜测”

    如果模型工具调用不稳定，再强的自然语言能力也很难落地。

    4. 成本和延迟

    把任务分层：

    层级任务推荐策略
    L1分类、改写、摘要快模型
    L2普通代码、结构化提取中等模型
    L3复杂推理、架构、Agent强模型

    不要让所有请求默认走最贵模型。生产系统通常要做路由：简单任务走快模型，失败或不确定时再升级到强模型。

    5. 数据与部署

    如果涉及客户数据、源代码、财务、医疗、合同，应优先确认：

    • 数据是否允许发到第三方 API
    • 是否需要私有化或本地部署
    • 是否有审计日志
    • 是否能做密钥隔离和访问控制
    • 是否满足所在行业的合规要求

    决策流程

    任务是否高风险？
├─ 是 → 先确认合规与人工审核，不直接自动化
└─ 否
   ├─ 是否需要长上下文？
   │  ├─ 是 → 长上下文模型 / RAG / 摘要压缩
   │  └─ 否
   ├─ 是否需要工具调用？
   │  ├─ 是 → Agent 友好模型
   │  └─ 否
   ├─ 是否对延迟敏感？
   │  ├─ 是 → 快模型优先，必要时降级质量
   │  └─ 否 → 质量优先

    最小评测集

    选模型前，先做一个 30-50 条的小评测集。每条包含：

    • 输入
    • 理想输出
    • 禁止行为
    • 判断标准
    • 真实业务样例

    示例：

    - id: code-review-001
  task: 审查一段 Express 登录接口
  must_find:
    - 密码错误不应暴露用户是否存在
    - 缺少限流
  must_not:
    - 编造不存在的中间件
  score:
    correctness: 0-5
    usefulness: 0-5
    format: 0-2

    常见错误

    • 只看榜单：榜单任务和你的业务任务通常不同。
    • 只测一次：模型输出有波动，要多样本、多轮测试。
    • 忽略失败成本：客服、财务、代码自动修改这类场景，错一次代价很高。
    • 不做路由：所有任务都走一个模型，成本和质量都不稳。
    • 不记录版本：模型升级后行为变化，必须记录模型名、版本、提示词和评测结果。
