LLM 评估与可观测性

把“感觉还行”的模型输出变成可测、可追踪、可迭代的工程系统。

更新于 2026年7月5日 评估可观测性LLMOps质量
目录

    为什么需要评估

    LLM 应用最危险的地方，是演示时很好，真实用户一多就暴露问题：

    • 输出格式偶发错误
    • RAG 引用不准确
    • Agent 调错工具
    • 成本突然升高
    • 模型升级后行为变化
    • 用户问法变化导致质量下降

    没有评估和观测，就无法判断一次改动到底是变好了还是变坏了。

    评估对象

    对象关注点
    Prompt是否稳定触发期望行为
    模型质量、成本、延迟、稳定性
    RAG召回、重排、引用、忠实度
    Agent工具选择、任务完成率、循环控制
    产品体验用户满意度、转人工率、重试率

    离线评估

    离线评估是在上线前用固定数据集测试。

    最小数据集

    每条样例包含：

    - id: rag-001
  input: "公司年假规则是什么？"
  expected_points:
    - "满一年可休年假"
    - "不同工龄对应不同天数"
  forbidden:
    - "编造不存在的假期"
  source_required: true

    评分维度

    维度说明
    正确性是否回答了事实
    完整性是否覆盖关键点
    忠实度是否基于证据，不编造
    格式是否符合 schema
    安全是否泄露敏感信息
    可用性是否对用户有帮助

    自动评分和人工评分

    自动评分适合：

    • JSON schema 校验
    • 字段完整性
    • 引用是否存在
    • 关键词覆盖
    • 正则规则

    人工评分适合：

    • 复杂业务判断
    • 语气和表达
    • 方案优劣
    • 代码可维护性

    不要完全依赖“让另一个模型打分”。LLM Judge 可以提高效率，但关键样本仍要人工抽检。

    在线观测

    上线后至少记录：

    指标用途
    请求量了解使用规模
    成功率判断系统稳定性
    首 token 延迟影响体感速度
    总延迟影响任务完成
    输入 / 输出 token成本核算
    模型名和版本复盘行为变化
    Prompt 版本对比不同提示词
    工具调用次数发现 Agent 异常
    用户反馈质量闭环

    日志结构

    建议每次 LLM 调用记录一个 trace：

    {
  "trace_id": "tr_123",
  "user_id": "u_001",
  "feature": "kb_qa",
  "model": "model-name",
  "prompt_version": "kb_qa_v3",
  "input_tokens": 1200,
  "output_tokens": 380,
  "latency_ms": 2400,
  "tool_calls": ["search_docs"],
  "status": "success"
}

    敏感内容要脱敏。日志不是越全越好，而是要能复盘问题，同时不制造隐私风险。

    RAG 专项指标

    指标含义
    Recall@K正确文档是否被检索到
    MRR正确结果排名是否靠前
    Context Precision塞入上下文的片段是否相关
    Faithfulness回答是否忠于证据
    Citation Accuracy引用是否指向真实来源

    RAG 问题要拆开看：是没检索到、检索到了但没排前、上下文拼接坏了，还是模型没按证据回答。

    Agent 专项指标

    指标含义
    Task Success Rate任务完成率
    Tool Accuracy工具选择是否正确
    Retry Count是否频繁失败重试
    Loop Count是否陷入循环
    Human Intervention Rate需要人工接管的比例
    Rollback Rate自动改动被回滚的比例

    Agent 不只看最终回答，还要看过程是否可控。

    发布策略

    1. Prompt 版本化

    每次改 Prompt 都标版本：

    support_rag_v1
support_rag_v2
support_rag_v3

    不要在生产里直接覆盖旧 Prompt。

    2. 小流量灰度

    先让 5%-10% 流量走新模型或新 Prompt，对比：

    • 用户满意度
    • 错误率
    • 成本
    • 延迟
    • 人工接管率

    3. 保留回滚

    模型、Prompt、检索参数、工具 schema 都要能回滚。LLM 应用的回滚速度很重要。

    常见错误

    • 只看平均分：长尾失败可能更关键。
    • 只测理想问题：真实用户会问错、问偏、问一半。
    • 不记录 Prompt 版本：无法复盘质量变化。
    • 不拆 RAG 链路：不知道问题出在检索还是生成。
    • 不算成本：功能越受欢迎，成本风险越大。
