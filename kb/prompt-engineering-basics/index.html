Prompt 工程完全指南

从定义到实战，系统掌握 Prompt 的核心原则、设计框架、场景技巧与进阶策略。

更新于 2026年7月5日 Prompt工程框架实战

    一、什么是 Prompt

    Prompt（提示词）是你与大型语言模型（LLM）交互的输入文本。它可以是简单的一句话，也可以是包含角色、上下文、示例和约束的复杂指令。Prompt 的质量直接决定了模型输出的质量——同样的模型，不同的 Prompt 可能产生天差地别的结果。

    为什么 Prompt 如此重要？

    • 模型不会读心：LLM 基于你提供的信息生成回复，模糊的输入必然导致模糊的输出。
    • 可控性的核心杠杆：在模型能力确定的前提下，Prompt 是你唯一能直接调控的变量。
    • 成本效率：一个精准的 Prompt 一次就能得到可用结果，省去反复调试的时间和 token 消耗。

    二、基础原则

    写好一个 Prompt，本质是把你脑中的意图无歧义地传达给模型。

    1. 清晰具体

    把”帮我写点东西”换成”用 100 字、面向初学者、解释什么是向量数据库”。

    反例：帮我写个产品介绍。

    正例：为面向 B 端客户的 SaaS 项目管理工具写一段 200 字的产品介绍，语气专业但不生硬，突出「多人协作」和「自动化报表」两个卖点，目标读者是中小企业的运营负责人。

    2. 给出结构

    明确角色、任务、输入、输出格式，让模型知道”你是谁""要做什么""怎么做”。

    角色：你是一名资深技术编辑
任务：把下面的草稿改写得更简洁
输出：Markdown 格式，分点列出，每点不超过 2 行

    3. 提供示例（Few-shot）

    给一两个”输入→输出”的范例，比长篇描述更有效。示例让模型直接从模式中学习，而不仅仅是理解文字描述。

    将以下产品功能转换为用户利益点：

示例：
输入：支持 256 位 AES 加密
输出：你的数据享受银行级安全保护

输入：API 响应时间 < 50ms
输出：系统操作几乎零等待，体验丝滑流畅

输入：支持多语言界面
输出：

    4. 设定约束

    字数、语气、禁止项，越明确越稳定。约束不仅是”不要做什么”，更是为输出划定边界。

    约束：
- 不超过 300 字
- 不使用"革命性""颠覆"等夸张词汇
- 不提及竞品名称
- 必须包含一个具体数据或案例

    三、通用 Prompt 设计框架

    以下框架适用于大多数场景，按需裁剪即可。

    CRISP 框架

    要素说明示例
    Context（上下文）背景信息，让模型理解当前场景”我们是一个面向大学生的在线编程教育平台”
    Role（角色）指定模型扮演的身份”你是一位有 10 年经验的 Python 讲师”
    Instruction（指令）具体要做什么”为以下知识点设计 3 道由浅入深的练习题”
    Specification（规格）输出格式、约束条件”JSON 格式，每题包含题干、选项、答案、解析”
    Parameters（参数/示例）参考样例或额外输入”参考以下样题风格：…”

    通用模板

    # 上下文
{背景描述，让模型理解当前场景}

# 角色
你是一名{领域}{资历}的{角色}。

# 任务
{具体要完成的事情}

# 要求
- 输出格式：{Markdown / JSON / 表格 / ...}
- 字数限制：{N 字以内 / 不限}
- 语气风格：{专业 / 轻松 / 严谨 / ...}
- 禁止事项：{不要...}

# 示例
输入：{示例输入}
输出：{示例输出}

# 输入
{实际要处理的内容}

    四、不同场景的应用技巧

    4.1 内容创作

    关键：明确受众、语气和目的，避免模型输出泛泛而谈的内容。

    角色：你是一名科技媒体主编
任务：为一篇关于 RAG 技术的文章写 200 字导语
要求：
- 读者是产品经理和运营人员，不是工程师
- 先抛出痛点（信息过载但找不到有用的），再引出 RAG 的价值
- 语气：专业但不学究，像在跟同事聊天
- 禁止使用"革命性""颠覆"等词

    4.2 数据提取与结构化

    关键：严格定义输出 Schema，提供示例，避免模型自由发挥。

    任务：从以下文本中提取人物信息，严格按 JSON 格式输出。

输出 Schema：
{
  "name": "string",
  "title": "string",
  "company": "string",
  "key_achievements": ["string"]
}

示例：
输入："张伟在字节跳动担任算法总监，主导了推荐系统升级，DAU 提升 30%。"
输出：{"name": "张伟", "title": "算法总监", "company": "字节跳动", "key_achievements": ["主导推荐系统升级，DAU 提升 30%"]}

输入：{待处理文本}

    4.3 代码生成与调试

    关键：说明技术栈、约束条件和边界情况，而非只描述功能。

    角色：你是一名 Go 后端工程师
任务：实现一个 HTTP 中间件，功能是对单个 IP 的请求做令牌桶限流
要求：
- 使用 Go 标准库，不引入第三方包
- 支持通过环境变量配置桶容量和填充速率
- 写单元测试，覆盖正常通过、限流拒绝、桶恢复三个场景
- 代码注释用中文
- 不需要 main 函数，只需要中间件和测试

    4.4 分析与决策

    关键：要求模型展示推理过程，而非只给结论。

    角色：你是一名产品顾问
任务：分析以下两个方案各自的优劣，并给出推荐
要求：
- 用表格对比两个方案在"开发成本""上线速度""可扩展性""用户体验"四个维度的表现
- 每个维度给出 1-5 分的评分和一句话理由
- 最后给出推荐方案和理由
- 先列出你的分析过程，再给结论

    4.5 多轮对话与教练式引导

    关键：设定行为规则而非具体回复，让模型在规则内自主决定每轮说什么。

    角色：你是一名对话教练
规则：
- 一次只问一个简洁、开放的问题
- 不替用户做决策，不直接给建议
- 优先使用用户自己的词汇和表达
- 如果用户请求建议，先征得允许，再把建议作为可选项呈现
- 当检测到用户可能处于危机状态时，优先引导寻求专业帮助

    五、进阶策略

    5.1 思维链（Chain of Thought）

    要求模型”一步步想”，显著提升逻辑推理的准确率。

    请一步步分析以下问题，先写出推理过程，再给出最终答案。

问题：一家咖啡店月租金 2 万，原材料成本占营收 35%，
员工工资合计 3 万/月，其他固定支出 5000 元/月。
如果月营收 12 万，月利润是多少？

    5.2 自我检查（Self-Verification）

    让模型生成后再审视自己的输出，发现并修正错误。

    第一步：生成回答。
第二步：审查你的回答，检查以下要点：
  - 是否遗漏了用户的要求？
  - 数据和事实是否准确？
  - 是否存在逻辑矛盾？
第三步：如有问题，修正后输出最终版本；如无问题，直接输出。

    5.3 角色叠加

    用多个角色分工协作，比单一角色更全面。

    请分别以以下三个视角评估这个产品方案：
1. 技术负责人：关注可行性和技术风险
2. 产品经理：关注用户价值和市场竞争力
3. 财务分析师：关注 ROI 和成本结构

每个视角用 3-5 句话表达核心观点，最后用 2 句话做综合判断。

    5.4 渐进式精炼

    先让模型生成初稿，再针对性打磨，比一次要求完美更高效。

    第一轮：基于以下要点写一篇 500 字的博客初稿。
第二轮：将初稿改写得更有吸引力——开头用一个具体场景切入，每段加一句金句，结尾加一个行动号召。
第三轮：检查改写后的文章，确保没有夸大表述和虚假数据，修正后输出最终版。

    5.5 占位符与模板化

    将可变部分用占位符表示，方便批量调用和程序化使用。

    你是一名{industry}领域的资深文案。

任务：为以下产品写一段{length}字的{content_type}。

产品名：{product_name}
核心卖点：{key_selling_points}
目标受众：{target_audience}
语气：{tone}
禁止：{restrictions}

    六、常见误区

    误区正确做法
    把所有要求塞进一句话分段落、分模块，结构清晰
    只说”不要做什么”同时说清楚”要做什么”，正向指令比负向约束更有效
    过度依赖模型”理解暗示”关键要求显式写出来，不要假设模型能猜到
    一次 Prompt 想解决所有问题复杂任务拆成多步，每步聚焦一个目标
    示例与实际任务格式不一致示例的格式、语气、详细度应与期望输出完全匹配

    七、检查清单

    写完 Prompt 后，对照以下清单自查：

    • 角色是否明确？模型知道自己”是谁”吗？
    • 任务是否具体？模型知道”做什么”吗？
    • 输出格式是否清晰？模型知道”怎么交付”吗？
    • 约束是否完备？有没有遗漏的边界条件？
    • 是否需要示例？给一个范例能减少多少歧义？
    • 上下文是否充分？模型缺少哪些背景信息？
    • 能否拆成多步？一步到位和分步走哪个更可控？
