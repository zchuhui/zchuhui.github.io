## 从想法到产品

作为一名 AI Builder，我相信**最好的学习方式就是构建**。过去一年多，我用 LLM 从零做了几个小产品，有的活下来了，有的死了。这篇文章不聊宏大叙事，只聊真实经验——从一个模糊想法到可用的产品，我实际做了什么、踩了什么坑、以及哪些认知被反复验证过。

LLM 改变的不是"能不能做"，而是"值不值得做"。以前做一个功能可能要一周，现在可能一个下午就够。这意味着试错成本急剧下降，真正稀缺的能力变成了：**判断什么值得做，以及把做出来的东西迭代到真正好用**。

## 我的工作流

### 第一步：定义问题，而不是定义模型

拿到一个新想法时，我强迫自己先写一段话，回答三个问题：

1. **谁**会遇到这个问题？
2. 这个问题**多痛**？是"有点烦"还是"没有这个就干不了活"？
3. 现有的解决方案**差在哪里**？

模型选型是最后一步。大多数时候，最强的模型（Claude、GPT-4）先拿来验证想法；如果跑通了，再考虑降级到更便宜、更快的模型。过早纠结于"用哪个模型"是典型的过早优化。

### 第二步：用 Prompt 跑通核心链路

不要一上来就搭架构、写前端。我的标准做法是：

- 开一个空白的 `.md` 文件，手动写出理想的输入输出样例。
- 在模型 Playground 里反复调 prompt，直到输出质量稳定。
- 用脚本跑 10-20 个真实 case，肉眼看结果。

这一步的关键是**快速验证这个任务的 LLM 可行性**。有些任务天然不适合当前 LLM（比如需要精确计算的、需要实时数据的、幻觉容忍度极低的），早发现比晚发现省一个月。

### 第三步：建立评测集，数据驱动迭代

一旦核心链路跑通，马上建评测集。我的原则：

- **评测集不用大，但要有代表性**。20-50 条精心挑选的 case 比 200 条随机 case 更有用。
- **覆盖边界情况**：空输入、超长输入、多语言混杂、恶意输入。
- **先定性再定量**：先人工看一遍所有输出，标记好坏，再考虑自动化评分。

自动化评测我一般分三层：
- **规则层**：格式校验、长度检查、关键词包含/排除——这些是硬性约束，不满足直接打回。
- **模型层**：用更强的模型做 LLM-as-Judge，评价相关性、准确性、有用性。
- **人工层**：定期抽检，校准自动评分的偏差。

评测集是产品迭代的北极星。没有它，你改了一版 prompt 根本不知道是不是真的变好了。

### 第四步：工程化落地

Prompt 调好了，评测通过了，接下来是真正让用户能用。这个阶段容易忽略的点：

**缓存策略**。LLM 调用又慢又贵，能缓存的尽量缓存。语义缓存（用 embedding 相似度匹配历史结果）比精确匹配缓存更实用，但要注意相似度阈值——设太高命中率低，设太低质量下降。

**降级与兜底**。API 会挂，模型会抽风。永远准备一个 fallback：可以是规则兜底、可以是缓存结果、可以是另一个厂商的模型。用户不在乎你用的是 GPT 还是 Claude，用户只在乎能不能用。

**可观测性**。至少记录三样东西：每次调用的延迟（P50/P95/P99）、Token 消耗、输出质量评分。这些数据是你做决策的依据——什么时候该换模型、什么时候该加缓存、哪里是瓶颈。

**流式输出与用户体验**。LLM 生成慢是常态，流式输出不是可选项而是必选项。但要处理好中断逻辑——用户点取消、切换页面、网络断开，这些场景都要有对应的状态管理。

## 一些心法

> 模型会变，但对问题的理解和对用户的同理心不会过时。

### 关于 Prompt

Prompt 是你的产品接口，要像写 API 文档一样认真对待。一个成熟的 prompt 应该包含：

- **角色与目标**：模型在扮演什么角色，目标是什么。
- **输入输出格式**：明确的 schema，最好给 example。
- **约束与边界**：不要做什么，比要做什么同样重要。
- **Few-shot 示例**：2-3 个高质量示例远超一长串文字说明。

另外，**把 prompt 纳入版本管理**。Prompt 的改动就是产品逻辑的改动，值得写 commit message。

### 关于模型选择

不要追求"最强模型"。考虑三个维度：质量、延迟、成本。你的产品定位决定了这三个维度的权重。实时对话产品延迟优先，离线批处理成本优先，医疗法律场景质量优先。

一个实用的策略是**模型分层**：简单任务用小模型（Haiku、Flash），复杂任务用大模型（Sonnet、GPT-4），极端 case 才上最强模型（Opus、o1）。用路由逻辑把请求分发给合适的模型，能省不少钱。

### 关于 RAG

RAG 不是银弹。它解决的是"模型不知道私有知识"的问题，但引入了一堆新问题：

- **检索质量**：chunk 大小怎么定？用什么 embedding 模型？要不要 rerank？
- **上下文窗口**：检索回来的文档塞得下吗？塞进去了模型真的看了吗？
- **引用准确性**：模型引用的内容真的在文档里吗？

我的建议：先用最简单的 RAG（固定 chunk + 向量检索 + 拼到 prompt 里）跑起来，根据实际效果再加复杂度。大多数场景不需要 HyDE、不需要 GraphRAG、不需要 Agentic RAG。

### 关于 Agent

Agent 的本质是给模型一个循环：思考→行动→观察→再思考。这很强大，但也很危险。

- **循环控制**：永远设最大步数，永远设超时。模型可能陷入无限循环。
- **权限边界**：给工具的权限越小越好。不要让模型直接操作数据库或文件系统，加一层沙箱或审批。
- **可回滚性**：Agent 的操作最好可逆。发出去的钱回不来，但改了的数据库记录最好有审计日志。

### 关于评测

前面提到了评测的技术方法，这里再补一条心法：**评测是手段，不是目的**。不要让"刷评测分数"变成你的目标。高分不等于好产品。真正的好产品指标是用户留存——用户用了还想用，这才是唯一不会骗你的数字。

## 踩过的坑

### 坑一：过度依赖模型能力

LLM 很聪明，但不是万能的。我做第一个产品时，试图让模型做价格计算和汇率转换，结果偶尔出现离谱的幻觉。后来我把计算逻辑抽出来用代码实现，模型只负责判断"用户想做哪种计算"，准确率从 85% 飙到 99%。

**教训**：模型擅长理解和生成，不擅长精确计算。让模型做它擅长的，代码做代码擅长的。

### 坑二：忽视输入清洗

用户输入什么都有可能。SQL 注入式的 prompt 攻击、超长文本撑爆上下文、各种奇怪的特殊字符……上线第一周我的日志里全是 surprise。后来加了一层输入处理：长度截断、特殊字符转义、敏感信息脱敏、注入检测，问题少了一大半。

### 坑三：评测集和线上分布不一致

我花了很多时间建了评测集，上线后用户反馈却一般。复盘发现我的评测 case 都是"理想输入"——格式规整、表达清晰、需求明确。但真实用户的输入是碎片化的、多轮追问的、中英混杂的。后来我从线上日志里采样补了一批真实 case，评测的指导意义才真正体现出来。

### 坑四：忽视了非功能需求

产品能用之后，我急着加新功能，忽略了速度优化和错误处理。结果用户反馈最多的不是"缺某个功能"，而是"太慢了"和"偶尔报错"。把 P95 延迟从 8 秒降到 3 秒，对用户留存的影响远超加三个新功能。

## 工具推荐

日常使用的一些工具：

- **模型调用**：各家的 API + LiteLLM 做统一代理，方便切换和 fallback。
- **Prompt 管理**：最开始就是 Git 里的 `.md` 文件，量大了之后用 LangSmith / Agenta 这样的平台。
- **评测**：简单的用 Python 脚本 + pandas 统计，复杂的用 Braintrust / Langfuse。
- **可观测性**：Langfuse / LangSmith 都行，关键是用起来，而不是搭完就忘了看。
- **RAG**：LlamaIndex / LangChain 的抽象很好，但如果你只做简单的 RAG，直接调 embedding API + 向量数据库更灵活。

## 最后

用 LLM 构建产品这件事，技术门槛已经低到史无前例。真正拉开差距的，是你对问题的理解深度、你对用户的共情能力、以及你把"能跑"迭代到"好用"的耐心。

模型会变，框架会换，但这些底层能力，十年后依然有效。
