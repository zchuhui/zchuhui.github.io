## 什么是上下文工程

上下文工程（Context Engineering）是围绕模型输入做的系统设计：选择哪些信息、按什么结构、在什么时机、以多少 token 交给模型。

Prompt 工程关注“怎么说”。上下文工程关注“给什么”。在真实系统里，后者往往更决定质量。

## 上下文的组成

一个完整上下文通常包含：

| 组成 | 作用 |
|------|------|
| 系统指令 | 定义角色、边界、输出规范 |
| 用户目标 | 当前要完成的任务 |
| 项目规则 | 技术栈、目录结构、团队约定 |
| 相关资料 | 代码、文档、数据库记录、检索结果 |
| 历史状态 | 已做过什么、失败过什么 |
| 工具结果 | Shell、API、浏览器、MCP 返回 |
| 输出格式 | JSON schema、Markdown 模板、表格 |

## 核心原则

### 1. 相关性优先

不要把“可能有用”的内容全塞进去。上下文越大，模型越容易被噪声干扰。

更好的做法：

```text
目标：修复登录接口 500
上下文：
- 登录路由文件
- AuthService
- User 模型
- 最近一段错误日志
- 统一错误处理函数
```

而不是把整个后端目录全部贴进去。

### 2. 结构化优先

模型更容易理解有结构的信息：

```markdown
# 当前目标
修复登录接口空指针

# 相关文件
- backend/api/auth.go
- backend/service/auth.go

# 约束
- 不修改响应格式
- 必须补测试

# 验证命令
go test ./...
```

### 3. 新信息覆盖旧信息

多轮任务里，旧计划很容易和新要求冲突。上下文里要明确：

```text
以下是最新约束，优先级高于之前所有讨论：
只修改知识库 Markdown，不改页面组件。
```

### 4. 工具结果要压缩

工具返回可能很长。日志、搜索结果、数据库查询结果都应该摘要后再进上下文：

- 保留错误堆栈顶部和根因
- 保留命中的文件路径和行号
- 删除重复日志
- 删除无关字段

## 常见上下文策略

### 1. 直接注入

适合短任务，把相关文件和要求直接放进 Prompt。

优点：简单、稳定。  
缺点：上下文容易超限。

### 2. RAG 检索

适合知识库、文档问答、客服系统。先检索，再把 Top-K 片段塞给模型。

关键点：

- 分块要符合文档结构
- 检索要支持元数据过滤
- 生成答案要引用来源
- 检索不到时要明确说不知道

### 3. 摘要记忆

适合长会话和 Agent。把历史对话压缩成状态：

```markdown
# 已完成
- 新增 OpenClaw 工具文章
- 修复工具文章站内链接

# 待做
- 丰富知识库模块

# 重要约束
- GitHub Pages base 为 /
```

### 4. 文件索引

适合代码库问答。先建立文件结构、符号、依赖关系，再按任务拉取相关文件。

### 5. Tool-first

适合 Agent。不要让模型凭记忆回答，而是先调用工具确认：

```text
先读取 package.json 和当前路由文件，再给出修改计划。
```

## 上下文预算

可以按比例分配 token：

| 部分 | 建议占比 |
|------|------|
| 系统和规则 | 10% |
| 用户目标 | 5% |
| 相关材料 | 55% |
| 工具结果 | 20% |
| 输出空间 | 10% |

输出空间经常被忽略。如果输入塞太满，模型没有足够空间输出完整答案。

## 质量检查

每次设计上下文，问 5 个问题：

1. 当前目标是否明确？
2. 哪些信息是必须的？
3. 哪些信息可能误导模型？
4. 旧上下文是否和新要求冲突？
5. 模型输出后如何验证？

## 反模式

- **整库粘贴**：噪声多、成本高、容易误判。
- **只给报错不给代码**：模型只能猜。
- **只给代码不给业务规则**：实现容易跑偏。
- **把密钥放进上下文**：安全事故。
- **让模型记住隐含规则**：应该写进规则文件。
